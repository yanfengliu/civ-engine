# Review Summary

The iter-1 fixes hold up well overall: poison/recover semantics, tick-numbering invariants, snapshot v5 fidelity, defensive copies on `getDiff`/`getEvents`/`getByTag`, behavior-tree preemption clearing, and the unified `submit()` pipeline all check out against the current code. Code quality has clearly improved, and the test suite has gained focused coverage for each fix. The two biggest residual risks I found are: (1) a real correctness bug in `findNearest` — the loop bound is Chebyshev (`max(width, height)`) but the underlying `getInRadius` uses Euclidean, so entities in the diagonal corner of any non-trivial grid can never be returned (the L11 concern from iter-1 was acknowledged but the fix never landed); (2) `processCommands` emits execution events outside its `try`/`catch`, so a buggy `commandExecutionListener` can throw past the fail-fast contract and leave the world partially mutated without the poison flag set. Several minor concerns are documented under Medium and Low.

# Iter-1 Regressions

### L11 not fixed — `findNearest` `maxRadius` is still Chebyshev for a Euclidean query
- **Iter-1 ID**: L11 (Opus, originally classified Low/Polish; severity should have been higher)
- **File**: `src/world.ts:528`, plus `src/spatial-grid.ts:114-150` for context
- **Problem**: The iter-1 review noted the Chebyshev-vs-Euclidean mismatch but said it would become a non-issue once H3's single-call fix was applied. Commit `27b775a` did apply the single-call fix, but `75dd6a2` reverted to the expanding-radius walk (correctly, for performance) — and the maxRadius still uses `Math.max(width, height)`. Because `SpatialGrid.getInRadius` filters by `dx*dx + dy*dy <= radiusSq` (Euclidean), the maximum reachable Euclidean distance is `Math.max(width, height)`, but the actual maximum entity distance on a `W×H` grid is `Math.hypot(W-1, H-1)`. On any non-trivial square grid (≥4×4), an entity at `(W-1, H-1)` searched from `(0, 0)` is at distance `√(2(W-1)²) > W`, so the loop completes with `bestId` still `undefined`. Concrete repro: `World({ gridWidth: 4, gridHeight: 4, ... })`, entity at `(3, 3)`, `world.findNearest(0, 0, 'position')` returns `undefined` even though the entity exists.
- **Why it matters**: Public API on the AI hot path silently returns "no entity" when one exists. AI agents using `findNearest` to locate the nearest target/threat will incorrectly conclude none exist whenever the search point and target are on opposite corners of any non-tiny grid. None of the existing `findNearest` tests cover the corner-to-corner case (they all use the 20×20 grid with sub-15 distances), so this is regressed but invisible in CI.
- **Suggested fix**: Set `maxRadius = Math.ceil(Math.hypot(this.spatialGrid.width - 1, this.spatialGrid.height - 1))` (or `... .width + .height` as a slightly looser but cheap bound). Add a regression test: 4×4 grid, entity at `(3, 3)`, `findNearest(0, 0)` returns the entity.

# High

### Listener exceptions during `processCommands` bypass the fail-fast/poison contract
- **File**: `src/world.ts:1471-1543` (`processCommands`), `src/world.ts:1545-1573` (`dropPendingCommands`), `src/world.ts:1843-1858` (`emitCommandExecution`)
- **Problem**: `processCommands` calls `this.emitCommandExecution(...)` from four sites: (a) line 1476 in the missing-handler branch, (b) line 1501 inside the success try-block, (c) line 1511 inside the catch on handler throws, and (d) inside `dropPendingCommands` for every aborted command. Sites (a) and (d) are NOT inside any try/catch. If a registered `commandExecutionListener` throws synchronously (which user code is allowed to do — there's no documented or enforced "listeners must not throw" contract), the throw propagates up through `processCommands` → `runTick` → `executeTickOrThrow`. `runTick`'s outer `try` (line 1335) only has a `finally`, no `catch`, so `finalizeTickFailure` is never called: `this.poisoned` stays `null`, `this.lastTickFailure` is left at whatever its previous value was, `gameLoop.tick` is not advanced, and `currentDiff`/`currentMetrics` from the prior tick stay in place. The world has been partially mutated (commands before the listener throw ran their handlers, eventBus was cleared, dirty flags were cleared) but is not flagged as poisoned. Subsequent `step()` calls happily run a new tick on top of this inconsistent state.
- **Why it matters**: The whole point of the v0.4.0 "poisoned world" contract — and the prominent ARCHITECTURE.md / api-reference.md / changelog claims about fail-fast — is that any tick-pipeline failure marks the world as poisoned. A `commandExecutionListener` that throws is the easiest way for a user (or a misbehaving renderer/history adapter) to silently break that contract. The `tickFailureListener` path in `emitTickFailure` (line 1646-1650, called from `finalizeTickFailure`) has the same problem: if a `tickFailureListener` throws, the throw replaces the original `WorldTickFailureError` on its way out, the failure that was being reported is lost, and although `this.poisoned` was set before the emit, `finalizeTickFailure` itself does not complete (it never returns the cloned failure to the caller).
- **Suggested fix**: Wrap the listener iteration in a try/catch that swallows or, better, collects listener errors into a synthetic listener-phase `TickFailure` so the contract still holds. Concretely, `emitCommandExecutionResult` and `emitTickFailure` should isolate listener exceptions (e.g. log or accumulate them, never let them re-enter the pipeline). The diff-listener loop in `runTick` (line 1442-1460) already does this — apply the same pattern to the other two.

# Medium

### `recover()` does not drain commands or events queued during the failed tick
- **File**: `src/world.ts:618-623` (`recover`), `src/world.ts:1469` (`processCommands` calls `commandQueue.drain()`)
- **Problem**: When a tick fails, the failing tick's commands have already been drained and either executed, attributed `command_handler_threw`, or marked `tick_aborted_before_handler` with `dropped` audit entries. But any commands submitted *during* a failing tick — by an executed handler before the failure, or by a system before a system-phase failure — sit in the (now-undrained) `commandQueue.buffer`. After `recover()`, the next `step()` will run those commands as part of a new tick. There is no event communicating "these commands survived a poisoned tick"; from the audit trail it just looks like they were submitted in the new tick. Same is true for events emitted during the failing tick: `eventBus.clear()` runs at the *start* of each tick (line 1336), so events from a failed tick remain in `getEvents()` until the next `step()` clears them.
- **Why it matters**: AI-agent operators tracking "what got executed when" by reading the audit trail miss the fact that some queued commands straddle a failure. Recover-then-step gives a different command-execution ordering vs. recover-then-clear-then-step, and there's no easy way for the operator to know which side they're on.
- **Suggested fix**: Either (a) document explicitly in `ai-integration.md` that commands queued during a failed tick survive `recover()` and run on the next `step()`, and that `getEvents()` may show stale events between failure and recover; or (b) have `recover()` optionally take a `{ resetQueue?: boolean }` and clear `commandQueue.buffer` + emit `tick_aborted_before_handler` for every still-queued command (consistent with `dropPendingCommands` semantics).

### `EventBus.getEvents` falls back to a shared reference when `JSON.stringify` throws
- **File**: `src/event-bus.ts:3-10, 51-62`
- **Problem**: `deepCloneJsonOrShallow` catches a stringify failure and returns the original `value`. The comment claims "events flow through `assertJsonCompatible` at emit time elsewhere" — but `World.emit()` (`src/world.ts:669-671`) does NOT validate event payloads, and `EventBus.emit()` doesn't either. So if a user emits a payload with a circular reference or a `BigInt`/`Function`/`Symbol`, the deep-clone silently degrades to a live-reference share without warning. Mutating that returned event later writes through to the engine state.
- **Why it matters**: The whole point of the v0.4.0 H_NEW1 fix was to make `getEvents` return a defensive copy. The fallback creates a footgun where the contract holds in the common case but silently breaks for exactly the inputs most likely to need it.
- **Suggested fix**: Either (a) call `assertJsonCompatible(data, …)` inside `EventBus.emit` (matching the `assertJsonCompatible` in `ComponentStore.set`/`World.setState`), so non-JSON events are rejected at emit time; or (b) on stringify failure, throw a clear "event payload is not JSON-cloneable; engine cannot return a defensive copy" error; (c) at minimum, fix the misleading comment in `event-bus.ts:55-57`.

### Tick-failure listeners receive an unsafe deep clone (now ok), but `getLastTickFailure` does too — every read JSON-clones
- **File**: `src/world.ts:980-982` (`getLastTickFailure`), `src/world.ts:2228-2230` (`cloneTickFailure`)
- **Problem**: `getLastTickFailure()` runs `JSON.parse(JSON.stringify(failure))` on every call. A diff-listener or history recorder polling this between ticks (or inside the listener loop while debugging) pays the full clone cost per call, even though `failure` is immutable from the engine's perspective. The clone is the right default for safety, but the cost is non-trivial: `TickFailure` includes the full error stack trace. The cheaper option used elsewhere (e.g., `getDiff`, `getMetrics`) is fine here because diff/metrics are heavyweight; `lastTickFailure` is lightweight but called often by history/scenario tooling.
- **Why it matters**: Minor perf trap for AI-agent operators that loop on `world.getLastTickFailure()` checking for failures. Not urgent but worth a comment or a memoized clone.
- **Suggested fix**: Cache the cloned failure in a private field, invalidated whenever `lastTickFailure` is reassigned (or when `recover()` runs). Or just freeze the constructed `TickFailure` and return it directly — it's already JSON-safe by construction (line 1685-1706 asserts).

### Pathfinding pushes overcost neighbors onto the heap and into `bestG`
- **File**: `src/pathfinding.ts:127-143`
- **Problem**: After M3's fix, `current.g > maxCost` causes a `continue` instead of `return null` — correct. But the neighbor expansion at line 129-143 still does `bestG.set(neighborHash, newG); cameFrom.set(...); heapPush(...)` even when `newG > maxCost`. With an inadmissible heuristic and a tight `maxCost`, this can grow `open` and `bestG` linearly with the number of neighbors, plus `cameFrom` ends up containing back-pointers for nodes that never participate in any returned path.
- **Why it matters**: Memory and CPU waste on path queries that exceed `maxCost`. Not a correctness bug (the overcost nodes are skipped on pop) but defeats the early-out intent.
- **Suggested fix**: Add `if (newG > maxCost) continue;` before the `bestG.set` block. Combined with M3 this still preserves correctness for inadmissible heuristics — we just don't pollute the search structures with unreachable nodes.

# Low / Polish

### `cloneTickFailure` and `cloneTickDiff` use `JSON.parse(JSON.stringify(...))` instead of `structuredClone`
- **File**: `src/world.ts:2228-2237`
- **Problem**: Both functions are documented (or commented) as deep-clone helpers and both pay the full JSON serialization cost. `structuredClone` is available natively on Node 18+ (the engine's documented minimum) and is roughly 2-5× faster for JSON-shaped data; it also handles circular references safely. Since these helpers run inside `getDiff`/`getLastTickFailure`/`onDiff`-listener paths, every tick may run them.
- **Suggested fix**: Replace with `structuredClone(diff)` / `structuredClone(failure)`. Remove the JSON-safety comment (structuredClone doesn't need it) — but keep `assertJsonCompatible` on the *write* path so the structuredClone result is still serializable.

### `processCommands` `tick` parameter is computed twice
- **File**: `src/world.ts:1345-1346`
- **Problem**: `runTick` computes `const tick = metrics?.tick ?? this.gameLoop.tick + 1;` then passes it to `processCommands(tick)`. But `createCommandSubmissionResult` at line 1595 calls `this.getObservableTick()` which also reads from `currentDiff`/`currentMetrics`/`lastTickFailure`/`gameLoop.tick`. During the tick body these may diverge (currentMetrics.tick was set to `tick` already at line 1326, but currentDiff is from the previous tick). For commands submitted *during* a tick (by a handler or a system using `world.submit`), the result's `tick` field is `getObservableTick()` not the in-flight tick, which can be confusing.
- **Suggested fix**: Either (a) pass the in-flight tick through to `createCommandSubmissionResult` so the submitted command's result.tick equals the tick during which it was submitted, or (b) document explicitly that `result.tick` is the tick at which the submission was *recorded* (before the new tick begins).

### `addTagInternal` and `setMetaInternal` don't validate the entity is alive on snapshot load
- **File**: `src/world.ts:925-950` (deserialize), `src/world.ts:1221-1262`
- **Problem**: Snapshot deserialization calls `addTagInternal(entityId, tag)` and `setMetaInternal(entityId, key, value)` without checking `entityManager.isAlive(entityId)`. A malformed snapshot with tags/meta on a dead entity creates index entries pointing to nonexistent entities. When `createEntity()` later recycles that id from the freeList, the new entity inherits the dead entity's tags/meta.
- **Suggested fix**: In the deserialize loops at lines 925-950, guard with `if (!world.entityManager.isAlive(entityId)) continue;` (or throw — your call).

### `World.recover()` doesn't reset `nextCommandResultSequence`
- **File**: `src/world.ts:618-623`, `src/world.ts:255`
- **Problem**: After `recover()`, the next submitted command gets a sequence equal to whatever value `nextCommandResultSequence` was at when the failure happened. Combined with the M2-style "commands queued during the failed tick survive" issue above, sequences can have gaps (rejected/dropped commands consumed sequences that aren't audible anymore after recover clears `lastTickFailure`). Not a bug per se — sequences are monotonic, just not contiguous — but worth documenting.
- **Suggested fix**: Document the gap behavior in `commands-and-events.md`, or reset `nextCommandResultSequence` along with the rest of the recover cleanup.

### `findNearest` early-out comment is slightly misleading
- **File**: `src/world.ts:535-538`
- **Problem**: The comment says "any entity outside the next ring would be farther than the best we've already found." It's talking about Euclidean distance, but the loop variable `r` is being treated as a Chebyshev radius (max-component distance), so "outside the next ring" isn't quite the right mental model. The early-out is mathematically correct (per the analysis in iter-1 H3) but the explanation conflates two distance metrics. Worth tightening once the L11 fix lands.
- **Suggested fix**: Change to "every entity returned by `getInRadius(cx, cy, r)` has Euclidean distance ≤ r, so any entity newly admitted at iteration r has Euclidean distance > r-1; if our best is already within (r-1)² we can stop."
