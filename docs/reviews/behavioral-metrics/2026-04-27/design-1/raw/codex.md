- [HIGH] `docs/design/2026-04-27-behavioral-metrics-design.md:97,118,167,320` The accumulator contract and the chosen built-in state shapes disagree. `observe` is specified as immutable (“must return new state, not mutate”), but the proposed states are `number[]` and `Map<string, number>`. Honoring that contract forces clone-per-bundle behavior (`[...state, x]`, `new Map(state)`), which directly fights the one-pass / low-memory story in §5.2 and §11. Either allow in-place mutation of private accumulator state or change the contract/state guidance.

- [HIGH] `docs/design/2026-04-27-behavioral-metrics-design.md:215-250,311-313,441-442` The public typing is too weak for the API this spec claims to offer. `runMetrics(..., Metric<unknown, unknown>[]) => MetricsResult` and `compareMetricsResults(...) => MetricsComparison` erase every metric result to `unknown`, yet the example dereferences `comparison.sessionLengthStats?.p95` and the acceptance criteria require “full TypeScript types.” Related problem: `compareMetricsResults` assumes recursively diffable plain data, but `Metric<TResult>` is unconstrained. If the helper stays in v1, the metric collection/result types need to preserve names and result shapes and likely constrain results to JSON-like/plain data.

- [HIGH] `docs/design/2026-04-27-behavioral-metrics-design.md:169,258,307,422` Empty-corpus `Stats` returning `NaN` is incompatible with the spec’s own JSON baseline workflow. `NaN` is not JSON-stable (`JSON.stringify` turns it into `null`), so a result produced in memory does not round-trip through `baseline-metrics.json` unchanged. That breaks the determinism/comparison story for the exact no-data case the spec is trying to handle gracefully.

- [MED] `docs/design/2026-04-27-behavioral-metrics-design.md:112,124,265,427` The `orderSensitive` story over-promises what an `Iterable`-only reducer can know. `runMetrics` cannot soundly detect whether an arbitrary iterable has a stable order, and `Map.values()` is a bad example because Map iteration order is deterministic by insertion. The same abstraction leak shows up in §16’s `bundles.filter(...)` guidance, which is not available on generic iterables.

- [MED] `docs/design/2026-04-27-behavioral-metrics-design.md:148-195`, `src/session-bundle.ts:46,119`, `src/world.ts:135,150` The built-in set omits the most obvious engine-generic outcome metrics already present in the bundle format: command acceptance/rejection and execution success/failure. V1 can tell you that command volume or type mix changed, but not that a new policy/build made 30% of submissions reject or executions fail. Those are bundle-derivable engine signals, not game semantics, so I would include at least one built-in around them.
