[HIGH] §6.10/§6.11 still mis-specify submission acceptance vs execution outcome. Repo contract says `bundle.commands` records every `submitWithResult(...)` result, including `accepted: false`, via `RecordedCommand.result` ([session-bundle.ts](/C:/Users/38909/Documents/github/civ-engine/src/session-bundle.ts:41), [session-recorder.ts](/C:/Users/38909/Documents/github/civ-engine/src/session-recorder.ts:165), [session-recorder.ts](/C:/Users/38909/Documents/github/civ-engine/src/session-recorder.ts:169)). `bundle.executions` is populated later from command-execution listeners only for queued commands that reach processing or are dropped after queueing ([session-bundle.ts](/C:/Users/38909/Documents/github/civ-engine/src/session-bundle.ts:109), [world.ts](/C:/Users/38909/Documents/github/civ-engine/src/world.ts:732), [world.ts](/C:/Users/38909/Documents/github/civ-engine/src/world.ts:1684)). Validator rejections therefore never appear in `bundle.executions`. As written, `commandAcceptanceRate` would over-report acceptance whenever validators reject commands, and the §6.10 note claiming `executed === false` covers validator rejections is false. Fix: define acceptance over `bundle.commands` (`accepted submissions / total submissions`) and keep `executionFailureRate` as an execution-stage metric over `bundle.executions`.
