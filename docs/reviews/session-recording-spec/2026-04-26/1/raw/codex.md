**Critical**
- The replay path is not implementable as written. `SessionBundle.commands` is only `CommandSubmissionResult[]`, and that type does not contain command payloads, yet `openAt()`/`selfCheck()` require resubmitting the original `type` and `data`. The same payload hole exists in the current `WorldHistoryRecorder`, so the `ScenarioRunner` path cannot recover it either. `docs/design/2026-04-26-session-recording-and-replay-design.md:97-103,335-339,352-358`; `src/world.ts:131-143`; `src/history-recorder.ts:41-45,171-174,194-203`
- The sink architecture does not compose with the current world listener contracts. The spec relies on awaiting sink writes and surfacing `SinkWriteError`, but `onDiff` listeners are synchronous and `onCommandResult`/`onCommandExecution`/`onTickFailure` listener failures are caught and logged by `World`, not propagated. An async `FileSink` cannot provide the promised backpressure or fail-fast behavior through these hooks. `docs/design/2026-04-26-session-recording-and-replay-design.md:252,258-277`; `src/world.ts:1542-1558,1729-1759`
- Failure/recover replay is fundamentally underspecified, and the current snapshot format cannot restore poisoned state. The spec claims a replayable record of “any `World` run” and explicitly records poisoned snapshots, but `WorldSnapshot` contains no poison/failure metadata, `serialize({ inspectPoisoned: true })` only suppresses a warning, and `openAt()`/`selfCheck()` never describe how recorded failures and later `recover()` calls are replayed. `docs/design/2026-04-26-session-recording-and-replay-design.md:251,337-340,353-358`; `src/serializer.ts:62-77`; `src/world.ts:641-646,841-847,1763-1782`

**High**
- The replay algorithm is off by one on command timing. `CommandSubmissionResult.tick` records the observable current tick at submission time, while queued commands execute on the next tick during `processCommands()`. Replaying commands whose submission tick equals `t` before stepping tick `t` is wrong; it also leaves command ordering within a tick implicit instead of sequence-defined. `docs/design/2026-04-26-session-recording-and-replay-design.md:337-339,353-357`; `src/world.ts:728-757,1275-1281,1564-1571,1674-1696`
- `selfCheck()` does not actually cover the whole timeline as specified. The bundle has a top-level `initialSnapshot` plus a separate `snapshots[]`, but the algorithm only iterates consecutive pairs inside `bundle.snapshots`. That skips the initial-to-first-snapshot span, and for scenario bundles as written, `snapshots` contains only one final snapshot, so `selfCheck()` becomes a no-op. `docs/design/2026-04-26-session-recording-and-replay-design.md:100,105,243,353-358,384`
- `ScenarioRunner` integration is built on the wrong current API surface. There is no `ScenarioRunner` class to add a method onto, only `runScenario(...)`; `ScenarioCheckOutcome` does not track a check tick despite the spec saying it already does; and the current scenario history is bounded by default, so adapting existing `ScenarioCapture.history` will silently truncate longer runs and produce non-replayable bundles. `docs/design/2026-04-26-session-recording-and-replay-design.md:368-390`; `src/scenario-runner.ts:44-48,104-113,133-139`; `src/history-recorder.ts:116-125`
- `fromSink()` and the on-disk format are underspecified for replay-critical data. The file layout only streams `ticks.jsonl` plus snapshots/attachments, even though replay also needs commands/executions/failures; meanwhile `SessionReplayer.fromSink()` accepts a write-oriented `SessionSink` interface that has no read/stream API beyond `toBundle()`, which is incompatible with the claimed lazy manifest/JSONL loading. `docs/design/2026-04-26-session-recording-and-replay-design.md:149-159,258-268,290-293`
- The determinism contract is incomplete because it ignores validator-time behavior. `submitWithResult()` runs validators immediately with a live `World`, before the command is queued, so “route all input through `world.submit()`” is not enough unless validators are also required to be deterministic and side-effect free. As written, a user can follow §11 and still create uncaptured out-of-tick mutations. `docs/design/2026-04-26-session-recording-and-replay-design.md:396-408`; `src/world.ts:728-757,760-766`

**Medium**
- Marker entity refs use bare `EntityId[]` even though the engine recycles ids and already has `EntityRef { id, generation }`. That makes `markersByEntity(id)` and any long-lived marker/viewer semantics ambiguous across entity reuse. `docs/design/2026-04-26-session-recording-and-replay-design.md:186-199,301`; `src/types.ts:1-5`; `docs/architecture/ARCHITECTURE.md:16,75`
- Historical marker validation is not grounded in the sink/recorder interfaces. §6.1 requires reconstructing liveness from “the most recent snapshot plus subsequent diffs,” but `captureInitialSnapshot` is optional and `SessionSink` exposes no readback API, so validating old markers on a streaming sink either requires keeping the full timeline in memory or using sink-specific escape hatches the spec does not define. `docs/design/2026-04-26-session-recording-and-replay-design.md:197-199,213-214,258-268`
- The documentation plan is incomplete relative to `AGENTS.md`. This is a new AI-facing standalone utility surface, so `docs/guides/ai-integration.md` and `concepts.md` should be in the required-update list, but §14 omits both. `docs/design/2026-04-26-session-recording-and-replay-design.md:467-492`; `AGENTS.md:115-118`
- Metadata/lifecycle is internally inconsistent. `SessionMetadata` includes `durationTicks` and `endTick`, but `connect()` calls `sink.open(metadata)` before those values exist; the comments also say `sessionId` is generated at `connect()` while the API section says it is generated at construction. `docs/design/2026-04-26-session-recording-and-replay-design.md:111-116,219,242-243`
- The testing section misses several failure modes that are central to this design: no tests for replay across recorded tick failures and `recover()`, no tests for validator-side effects, no tests proving `selfCheck()` covers the initial snapshot span / scenario bundles, and no tests for async sink behavior against the current listener contracts. `docs/design/2026-04-26-session-recording-and-replay-design.md:448-463`

**Low**
- No material low-severity findings. The problems here are architectural and contract-level, not polish.

**Note**
- ADR 1, keeping session recording separate from `WorldHistoryRecorder`, is defensible. The spec’s main weaknesses are elsewhere: the bundle is missing replay-grade inputs, the sink boundary does not match the engine’s listener semantics, and the “documented, not enforced” determinism story depends on a `selfCheck()` that currently skips key spans and failure cases.

Needs rework on replay input capture, failure/recover semantics, and the sink/replayer contract.
