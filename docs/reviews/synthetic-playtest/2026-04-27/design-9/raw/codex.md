But recorded bundles key commands by `submissionTick`, which is the observable pre-step tick, not the about-to-execute tick ([session-recording design](C:/Users/38909/Documents/github/civ-engine/docs/design/2026-04-26-session-recording-and-replay-design.md:130), [off-by-one rationale](C:/Users/38909/Documents/github/civ-engine/docs/design/2026-04-26-session-recording-and-replay-design.md:479), [recorder capture](C:/Users/38909/Documents/github/civ-engine/src/session-recorder.ts:455), [world submit tick](C:/Users/38909/Documents/github/civ-engine/src/world.ts:1803)). A bundle command captured at `submissionTick === 0` is processed by the step from `0 -> 1`; if a caller maps that directly into `ScriptedPolicyEntry.tick = 0`, the current spec makes that entry unreachable or shifts playback by one step. The spec should either define `ScriptedPolicyEntry.tick` in submission-tick terms and match against `world.tick`, or keep executing-tick semantics and explicitly require `submissionTick + 1` when deriving scripts from bundles, with a regression test for that conversion.
