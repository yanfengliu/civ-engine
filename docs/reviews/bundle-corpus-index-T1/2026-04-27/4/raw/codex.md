1. Medium - `BundleQuery` runtime validation is still inconsistent for malformed callers, which leaks raw behavior instead of the documented `CorpusIndexError(code: 'query_invalid')` contract. In `src/bundle-corpus.ts:294-299`, any non-string `key` is treated as a `RegExp`; in `src/bundle-corpus.ts:324-327`, a non-object `recordedAt` can silently no-op; in `src/bundle-corpus.ts:357-358`, any truthy `attachmentMime` is accepted. The public docs present these as validated query fields and document `query_invalid` as the machine-readable failure mode (`docs/api-reference.md:5235-5259`, `docs/api-reference.md:5280-5291`), but the tests only cover bad numeric ranges / bad ISO bounds inside correctly shaped query objects (`tests/bundle-corpus.test.ts:301-309`). For JS consumers or `as any` call sites, this can produce silent mis-filtering or a raw `TypeError` instead of the advertised corpus error surface.

2. Medium - The new suite does not pin the full public error taxonomy, even though the docs expose those discriminators as API. `manifest_parse` and `schema_unsupported` are distinct public codes in `src/bundle-corpus-types.ts:45-52`, with dedicated branches in `src/bundle-corpus-manifest.ts:181-196,224`, and they are documented in `docs/api-reference.md:5280-5291`. But the staged tests only exercise `query_invalid`, `entry_missing`, `root_missing`, and `manifest_invalid` (`tests/bundle-corpus.test.ts:301-342`). Since callers are expected to branch on `error.details.code`, leaving those paths untested makes it easy for a later refactor to collapse them into a generic invalid-manifest failure without the suite catching the contract regression.
