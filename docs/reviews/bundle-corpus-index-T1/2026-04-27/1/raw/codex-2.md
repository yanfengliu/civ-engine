Medium — `scanDepth: 'children'` has an observable contract mismatch between docs and code. `discoverBundleDirs()` always checks `rootDir` first and short-circuits if it finds `manifest.json`, so a root bundle returns only `'.'` instead of child bundles. That conflicts with the new public docs that describe `'children'` as “one directory level below the root” / “direct child directories”. Refs: `src/bundle-corpus.ts:129`, `docs/api-reference.md:5178`, `docs/guides/bundle-corpus-index.md:85`. There is also no test covering the root-is-itself-a-bundle case for `'children'`, so this behavior is currently undocumented and unpinned.

2. Medium — The new immutability contract is not type-sound. `BundleCorpusEntry.metadata` is exported as `Readonly<SessionMetadata>`, but `SessionMetadata.failedTicks` is still `number[]`, so TypeScript allows `entry.metadata.failedTicks?.push(...)` even though the runtime freezes that array and the tests expect a `TypeError`. Refs: `src/bundle-corpus-types.ts:75`, `src/bundle-corpus.ts:152`, `tests/bundle-corpus.test.ts:242`. For a new public API, this should be modeled as a truly readonly nested shape, not enforced only at runtime.

3. Low — `docs/api-reference.md` does not exactly match the shipped `loadBundle` signatures. The docs omit the generic constraints present in the real API, even though this file is supposed to be the authoritative public surface. Refs: `docs/api-reference.md:5161`, `docs/api-reference.md:5206`, `src/bundle-corpus-types.ts:82`.

I did not find a root export leak in package surface: `package.json` still exports only `"."`, so the module split itself does not appear to expose unintended subpaths.
