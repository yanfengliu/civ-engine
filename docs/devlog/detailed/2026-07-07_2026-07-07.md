# Devlog - 2026-07-07

## Shared improvement loop finding contracts (v1.4.0, additive minor)

**Timestamp:** 2026-07-07

**Action.** Added the first shared public API slice for the agent recursive self-improvement loop: `IMPROVEMENT_FINDING_SCHEMA_VERSION`, `ImprovementFinding`, `ImprovementEvidenceRef`, `ImprovementRunManifest`, `ImprovementVerificationStatus`, `ImprovementNextAction`, `ImprovementDisposition`, `improvementFindingToVisualPlaytestFinding`, `improvementFindingToMarker`, `improvementFindingsFromMarkers`, and `assertImprovementFinding`. The helper path converts loop findings to visual findings and records annotation markers carrying both `data.improvementLoop` and `data.visualPlaytest`; `getAiContractVersions()` now exposes the `improvementFinding` schema marker. Bumped engine/package version to 1.4.0 and updated README, API reference, AI Integration, visual harness guide, architecture, drift log, ADR 54, changelog, and public-surface fixture.

**Code reviewer comments.** External Codex and Claude CLI reviews were not run because the approval policy rejected exporting the staged unpublished diff to external reviewer services. Local adversarial review kept the engine boundary narrow and found one real issue: the new machine-facing finding schema was not exposed through `getAiContractVersions()`. Fixed by adding `IMPROVEMENT_FINDING_SCHEMA_VERSION`, the `improvementFinding` contract-version field, docs coverage, and a failing-first test. Review synthesis: `docs/threads/current/agent-recursive-improvement-loop/2026-07-07/1/REVIEW.md`.

**Result.** Game repos can now store durable recursive-loop findings with stable ids, plural evidence, verification status, next-action hints, optional disposition/source-run data, and game-specific JSON metadata while preserving visual-harness marker compatibility.

**Reasoning.** The broader loop design should become real by extracting proven common pieces, not by moving a whole game-specific controller into the engine. `aoe2` already proved the need for findings that survive visual/conformance reports, marker replay, and proposal-only workflows. A shared finding contract is the smallest useful cross-repo primitive; the rest of the loop remains local until more migrations prove it is genuinely reusable.

Green: `npx.cmd vitest run tests/improvement-loop.test.ts tests/public-surface.test.ts tests/visual-playtest.test.ts` passed (19 tests), and `npx.cmd tsc --noEmit` passed. Full gates passed: `npm.cmd test` (1254 passed + 1 todo), `npm.cmd run typecheck`, `npm.cmd run lint`, and `npm.cmd run build`. Dependency audit was not required because no dependency surface changed.

## Visual playtest harness contracts (v1.3.0, additive minor)

**Timestamp:** 2026-07-07

**Action.** Added a zero-runtime-dependency visual playtest harness surface for browser-game LLM loops: `runVisualPlaytestLoop`, visual observation/control/action/finding/trace types, hidden-state audience channels, prompt/redaction helpers, and session-marker conversion. Split public type vocabulary into `src/visual-playtest-types.ts` and runner/helpers into `src/visual-playtest.ts` to stay under the 500-line source budget. Updated root exports and the public-surface fixture, bumped engine/package version to 1.3.0, and documented the surface in README, API reference, docs hub, AI integration, AI playtester, architecture docs, drift log, ADR 53, changelog, and `docs/guides/visual-playtest-harness.md`.

**Code reviewer comments.** Codex CLI review was unavailable: the local PowerShell wrapper was blocked, and rerunning via `codex.cmd` with network access was rejected because it would upload the private workspace diff without explicit user approval. Claude CLI review was unavailable: sandboxed CLI returned `API Error: Unable to connect to API (ConnectionRefused)`. Local adversarial review found three real issues: nested `VisualPlaytestActionResult.observation` values bypassed safe trace redaction, `VisualPlaytestStateChannel.redaction` exposed a public field whose behavior was underdefined, and explicit `traceObservation: 'full'` still redacted the primary trace observation. All were fixed and pinned with tests. Review synthesis: `docs/threads/done/visual-playtest-harness/2026-07-07/1/REVIEW.md`.

**Result.** The engine now owns the shared visual playtest contract and generic loop while game repos keep browser automation, DOM/canvas control extraction, screenshots, provider clients, and game-specific hidden-state adapters. Hidden/debug state is explicit and audience-labeled (`agent`, `reviewer`, `traceOnly`), with player-blind vs oracle-assisted prompt behavior and safe trace redaction documented.

**Reasoning.** The game repos had converged on the same high-level visual loop but not on the browser/model adapter details. Pulling those dependencies into core would break the zero-runtime-dependency promise and disrupt active game development; leaving the contract informal would continue duplicating action/finding/redaction/trace glue. The additive core contract gives repos a shared target without forcing immediate migration.

**Validation.** Failing-first tests added in `tests/visual-playtest.test.ts` for loop ordering, multi-action decisions, explicit stop, max-step stop, host/agent/action failures, coded invalid `maxSteps`, hidden-state prompt modes, explicit redaction metadata, safe vs full trace observation capture, nested action-result redaction, and marker round-trip through `SessionRecorder`. Full gates were green after the review artifact landed: `npm test` (1249 passed + 1 todo), `npm run typecheck`, `npm run lint`, and `npm run build`.

**Notes.** No dependency sections changed, so the dependency-change audit protocol was not triggered. External multi-CLI review requires explicit permission to send diffs outside the local environment.

## Agent recursive improvement loop design (active design thread; no version bump)

**Timestamp:** 2026-07-07

**Action.** Added `docs/threads/current/agent-recursive-improvement-loop/DESIGN.md`, a cross-game design for turning the existing playtest/replay/corpus substrate into a full recursive improvement loop. The design defines the target loop (`run -> record -> find -> verify -> classify -> regress -> fix/propose -> review -> rerun -> compare -> learn`), shared game-adapter shape, run-manifest and finding expectations, evidence-reference requirements, ledger expectations, engine-vs-game responsibilities, current gap map, rollout phases, and success criteria.

**Code reviewer comments.** No code changed. Spec self-review checked for placeholders and internal consistency. No external multi-CLI review was run for this design-only artifact.

**Result.** The active thread now has an authoritative design doc for the cross-game harness objective. It frames the gap as uniformity, closure, and promotion rather than missing base engine capability: `civ-engine` already owns replay, visual-playtest contracts, markers, metrics, corpus, and inspection primitives; the missing layer is a portable controller contract that every game can implement.

**Reasoning.** The local engine-consuming games are at different maturity levels. `aoe2` has the richest local loop, `farm` has a smaller visual-loop path already aligned with `runVisualPlaytestLoop`, `city` has replay/self-check proof, and `civ-sim-web` / `idle-life` need minimum-loop bring-up. The design recommends proving one full vertical slice first, then extracting only the genuinely shared schemas/helpers back into the engine while keeping browser/provider/game-specific automation local.

**Validation.** `npx.cmd vitest run tests/docs-threads.test.ts` passed (2 tests). Placeholder scan for `TBD`, `TODO`, `FIXME`, and `???` in the design doc returned no matches.

**Notes.** No public API, runtime behavior, dependency, package version, changelog, or architecture boundary changed. The objective remains under `docs/threads/current/` until an implementation plan and review loop close it.
