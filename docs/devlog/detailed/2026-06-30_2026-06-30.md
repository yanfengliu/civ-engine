# Devlog — 2026-06-30

## Benchmark gate: add the `commands` (aoe2-shape) scenario — close the load-profile blind spot

**Timestamp:** 2026-06-30

**Action.** Added a fourth regression-gate scenario, `commands`, to `scripts/rts-benchmark.mjs` (+ `benchmarks/baseline.json`). It drives a STRICT world with per-tick external commands (a validator that rejects the no-op subset, handlers that mutate + emit inside the tick phase), an in-tick event listener, and 14 systems (12 querying across all five phases + a movement writer + a combat emitter) over 60 ticks on 800 entities. Deterministic (arithmetic ids/moves, count-based budgets, no wall-clock control flow), so the gate's tier-1 exact-counter assertion holds: `queryCalls 1560`, `queryResults 816000`, `commandsAccepted 3840`, `commandsRejected 960`, `commandsProcessed 8640`, `moveHandlerRuns 3840`, `eventsHandled 17400`. The `commandsProcessed` (engine-side) + `moveHandlerRuns` (handler-side) counters were added in review so a regression that drops/no-ops accepted `move`-handler execution fails the tier-1 exact-counter check rather than slipping through the time-only tier-2 ratio (a no-op handler is *faster*). No `src/` change — dev-tooling only, so no ENGINE_VERSION bump and no changelog entry.

**Why.** This was triggered by the co-evolving `aoe2` repo, whose `docs/engine-feedback/current.md` had recorded a "+50-75% sim-throughput regression" attributed to civ-engine 0.8.24→1.0.1 and asked the engine team to "profile a long mixed-load scenario against v0.8.23 and bisect." Investigation on the aoe2 side (see aoe2 `docs/debugging/2026-06-30-engine-throughput-regression.md`) proved there is NO engine hot-path regression: every per-tick hot-path file (`world-tick`, `world-queries`, `component-store`, `entity-manager`, `world-systems`, `query-cache`) is byte-identical across the whole 0.8.23→1.0.2 window; the only per-tick change is v1.0.0's strict-by-default flip (`world-core.ts`: `config.strict === true` → `!== false`), whose cost measured WITHIN NOISE on aoe2's real `ageUp` fixture (42.71s strict-on vs 43.24s strict-off, isolated). The observed +50-75% was full-suite thread-pool CONTENTION on aoe2's CPU-heavy sim suite, not engine cost.

The engine-feedback had also noted (correctly) that the gate "under-weights whatever aoe2's fixtures stress" — the existing `medium`/`large` scenarios are occupancy/pathfinding and `churn` is 20 ticks of entity spawn/destroy with NO commands, events, or strict mode. The `commands` scenario closes that blind spot so a FUTURE real per-tick regression in the strict-gate / command-queue / event-dispatch / many-system-per-tick path would be caught by the gate rather than surfacing downstream as a mysterious consumer slowdown.

**Validation.** `npm run benchmark:check` passes (all counters exact; `commands` timeRatio ~10 within the ×3 bound). Four gates green: typecheck, lint (`eslint src tests mcp/src mcp/tests`), build (`tsc -p tsconfig.build.json`), test (1238 passed + 1 todo). `benchmark-gate.test.ts` uses synthetic baselines, so the new scenario needs no test change. Determinism confirmed by the harness's own identical-counters-across-3-runs assertion.

**Notes.** aoe2 consumes the engine via `file:../civ-engine` (symlink); the scenario is a script-only addition so nothing in aoe2's runtime path changed. Multi-CLI reviewed alongside the aoe2 diff.
