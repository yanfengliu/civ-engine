## 2026-04-29 (process tightening) - AGENTS.md cleanup (v0.8.10)

**Action:** Removed the Tie-Breaker role and the hard-abort rule from AGENTS.md, moved the `docs/learning/lessons.md` rule into the Documentation discipline section, and trimmed redundancy between the Core-rules multi-CLI review mandate and the Code review section header/intro. Side-effect touches in the diff: version bump 0.8.9 → 0.8.10 across `package.json` / `src/version.ts` / README badge, new changelog + devlog entries, and a one-line regex change in `tests/docs-threads.test.ts` removing `tie-breaker` from the allowed iteration-folder name set (no historical thread used that folder name; verified via glob).

**Reasoning:**

- **Tie-Breaker removal.** The role specified `claude --model "claude-opus-4-7[1m]" --effort max` to break ties between engineer + reviewer. Same-family model breaking ties on its own family's review doesn't add genuine independence — the higher effort buys some delta but the structural blind spots are shared. The honest escalation path when LLM consensus stalls is "ask the human"; that's now what the Software-engineer role does after 3 iterations.
- **Hard-abort removal.** The "if tests fail or review consensus is not reached after 3 iterations, execute a hard abort + `git reset --hard` + write to `docs/learning/lessons.md`" rule was an artifact of a multi-developer workflow with merge gates. In the current solo-developer mode (commit directly to `main`, no PR gating), there's no benefit to mechanical aborts; in practice the agent should iterate or surface the disagreement.
- **`docs/learning/lessons.md` rule moved.** The file was only written by the (now-removed) hard-abort rule, even though it was *read* by re-reviewers per the iteration-folder rule. Without a write trigger, the file would silently rot. Moved to Documentation discipline → "Update if applicable to the change's topic" with a clearer trigger ("when you encounter a non-obvious failure mode worth preserving").
- **Redundancy trim.** The `Code review (mandatory; not optional)` section opened with: "This section is the operational implementation of the Core-rules 'Multi-CLI code review is mandatory' rule. Every code or behavior change runs through this loop before merge — no exceptions, no carve-outs for 'small' or 'single-file' changes." That re-states the Core rules paragraph word-for-word in places. The Core rules paragraph already carries the anti-rationalization armor (the explicit list of forbidden phrases).
