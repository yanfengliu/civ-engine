The engine provides the data structure; the game decides which layers to instantiate, what they contain, and how to tick them. This keeps the engine surface minimal and avoids forcing every game to pay for layers it doesn't use.

**Reasoning:** Of the three MicropolisCore ideas, this one had the longest design space: should the transaction buffer reads too (shadow view)? Should it pre-validate everything before applying or trust the underlying World mutations? Should it own a "snapshot, attempt, rollback on failure" model? I picked the pragmatic minimum:

- **No shadow view.** Reads inside a precondition or after commit see live world state. The user is responsible for not depending on their own buffered mutations during a precondition. Simpler to reason about, much smaller implementation, and matches the documented contract that "preconditions see the pre-commit baseline."
- **No snapshot/rollback.** Mutations apply through the existing public `World` API in registration order. If a mutation throws (e.g., entity destroyed between buffering and commit), the error propagates and the world is in a partially-applied state. Documented as a v1 caveat — caller should add a `require(w => w.isAlive(entity))` precondition for any entity it didn't itself just create.
- **Preconditions as the validation layer.** All "would this be allowed?" logic belongs in `require()` predicates, which run before any mutation applies. The transaction-layer code only validates "is this transaction still usable?" (single-use lifecycle). This keeps the transaction code small and pushes the policy decisions back into game logic where they belong.
- **v1 surface deliberately limited.** No `createEntity`/`destroyEntity` (entity ID allocation isn't rollback-safe — if a transaction allocated an entity then aborted, the ID would be lost forever). No tags/meta/state in v1 — those are smaller wins and easier to add incrementally if game code asks for them. Resources made the v1 cut because they're the primary motivating example (cost-checked actions).

**Notes:**

- Considered but rejected: making `CommandTransaction` extend or embed `CommandQueue`. The two abstractions answer different questions — `CommandQueue` is about deferring command processing to the next tick boundary and providing structured results to async callers; `CommandTransaction` is about gathering mutations into one atomic apply. Coupling them would force every transaction to round-trip through a tick, which defeats the synchronous "validate cost + apply now" use case.
- Considered but rejected: a richer `TransactionResult` failure type with a `details: JsonValue` payload (analogous to `TickFailure`). Premature — the current discriminated union with `code` and `reason` is enough for the documented use cases. Add structured details if a real game needs them.

If profiling shows transaction mutations dominate allocations in some game, the obvious next step is parallel typed arrays per mutation kind, not a different runtime model.
- Tests cover the propose-commit happy path, both precondition-failure paths (return `false` and return `string`), short-circuit on first failed precondition, atomicity (every original value unchanged on failure), the "preconditions see baseline" rule (precondition reads `hp` and gets the pre-buffer value, not the proposed-write value), the abort/commit lifecycle including double-commit-throws and chain-after-commit-throws, and the in-system "all mutations in one TickDiff" capture. The cost-check use case is tested in both directions (player has enough wood → build succeeds, building placed; player short → build aborts, wood unchanged, no building placed) since that's the primary motivation for the abstraction.
