## Why the engine is already 80 % of a lockstep engine

Classic deterministic lockstep needs exactly four properties: (1) identical simulation given identical inputs — the engine's documented determinism contract (design-1 wording fix: the contract is documented but NOT structurally enforced; `strict: true` structurally covers only the out-of-tick-mutation clause, and `selfCheck` is the offline verifier — in lockstep, the digest + diffBundles pairing below is the live-desync analogue of selfCheck); (2) a serializable, ordered input stream — `RecordedCommand` with `submissionTick` + `sequence` is precisely this, and bundles are its wire-format-ready container; (3) state snapshots for join/recovery — `WorldSnapshot` + `applySnapshot`; (4) divergence diagnosis — `diffBundles` + the registration manifest (v0.8.18) already diagnose the two dominant desync classes (input mismatch, registration drift) post-mortem. What is genuinely missing is the *coordination* layer and a *cheap online* desync probe.

## Architecture (designed now, built on demand)

### 1. Topology and trust

v1 targets **deterministic lockstep with a relay** (server relays inputs, simulates nothing) — the RTS-classic model that fits the engine's "every peer simulates everything" determinism. Server-authoritative simulation (server runs the world, clients render snapshots/diffs) already works today over `ClientAdapter` and needs no spec; the lockstep design exists for the bandwidth/scale profile where shipping inputs beats shipping diffs.

### 2. `LockstepSession` (future standalone utility — the only new runtime piece)

Transport-agnostic coordinator, one per peer, in the `ClientAdapter` mold (callbacks in, callbacks out, zero sockets in the engine):

- **Input frames, including empty ones (design-1 Codex/Claude):** every peer sends `InputFrame(tick, playerId, commands[])` for EVERY tick in its active range — `commands` may be empty. Mandatory empty frames are what make "no input this tick" distinguishable from "packet missing"; without them the barrier cannot be implemented correctly.
- **Tick barrier with input delay D:** commands submitted locally at tick T are scheduled for execution at T+D and broadcast immediately. The session steps the world to tick N only when an InputFrame for N is present from every member of the active roster at N (see membership). D hides network latency; the classic 2-5 tick window.
- **Deterministic cross-peer ordering:** all commands for tick N execute sorted by `(playerId, localSequence)` before `world.step()` — making peer arrival order irrelevant. This ordering rule is THE correctness-critical line of the whole design. **It binds only if each peer's session submits the sorted set to its LOCAL world in exactly that order** — submission order is execution order in this engine (FIFO queue; `submissionSequence` stamped at submit). Corollary footgun, stated loudly in the future guide: in lockstep mode, game code submits via the session, never `world.submit()` directly — one direct submit on one peer is an instant desync. (`SessionRecorder`'s ownership of `submitWithResult` is the in-repo precedent for a coordinator owning the submission path.)
- **Membership is relay-declared, tick-effective (design-1 IMPORTANT):** the active roster is versioned by **epochs**; joins AND leaves/timeouts are control messages issued by the relay (the single ordering point), each effective at a named future tick — so every peer agrees, deterministically, on the required-input set for every tick. Peers never infer departure locally (locally-inferred timeouts would desync the barrier's own membership set: peers would disagree about which tick a dead peer stopped counting for). The contract is half-open and exact: a membership change with `effectiveTick = T` applies to tick T and all later ticks — a leave with `effectiveTick = T` means the departing player's last required InputFrame is T-1 (the barrier for T no longer counts them), and a join with `effectiveTick = T` means their first required frame is T. One rule, applied identically to joins and leaves; the test plan's leave case pins it.
- **Stall policy:** waiting (pause-on-missing-peer) is v1 for transient gaps; permanent departure is handled by the relay's leave/timeout declaration above (waiting alone cannot be the whole policy — a crashed peer would stall the session forever). Speculative execution + rollback is explicitly rejected on three grounds: (a) rollback exists to mask input latency for human perception, and this engine's operators are AI agents that do not perceive a 2-5 tick delay — rollback's entire benefit is void here; (b) resimulation would re-emit events/diffs to `onDiff`/`getEvents` consumers, forcing speculation-awareness onto the whole observer surface; (c) per-tick rewind is *possible* via snapshots (`openAt`), but rollback netcode is a different genre of complexity and not what RTS-scale games need.
- **Join/rejoin:** snapshot transfer + `applySnapshot` + registration-manifest check (the v0.8.18 verifier runs verbatim on the joiner's factory — lockstep gets fail-fast "your build registers different systems" for free) + **version envelope (design-1 Claude; tightened in design-2):** the handshake exchanges `engineVersion` plus an exact runtime fingerprint — Node full version AND `process.versions.v8`. engineVersion follows the replayer's cross-`a`/cross-`b` reject policy (`BundleVersionError` precedent); the runtime check is deliberately STRICTER than the replayer precedent (which only warns on Node-major skew) and the spec owns that as a tightening: networked peers on separately-deployed builds are more exposed than record/replay on one machine, "same Node major" does NOT guarantee same V8, and only an exact V8 match makes unspecified-precision game-code floats (transcendental math the determinism contract does not cover) agree by construction. Peers may opt down to warn-only at their own risk.

### 3. Desync detection: cheap online probe + rich post-mortem

- **Online:** every K ticks (e.g., 30), each peer computes a **state digest** and piggybacks it on its input packet; mismatch → structured desync event naming the first divergent tick range. The digest is the one genuinely new engine primitive this design needs: `world.stateDigest(): string` — a stable hash over the serialized state. v1 digest = FNV-1a over `JSON.stringify(serialize())` (the engine already guarantees deterministic serialization order — component stores iterate in registration order — which for COMPONENTS is a documented factory obligation (ADR 4: register in the same order), not structurally pinned (the registration-manifest check order-compares systems but checks components extras-only) — so `stateDigest` canonicalizes by sorting component keys before hashing rather than trusting call order; integer-keyed records stringify in ascending numeric order per the JS spec, and the entity free-list is explicit in the snapshot so ID-recycling order survives `applySnapshot`); computed at a defined post-step barrier point. A 64-bit FNV variant is nearly free (32-bit ≈ 2⁻³² collision odds per comparison; real desyncs compound, so a collision only delays detection by one K-window). O(world) per call but amortized by K. An incremental digest (updated from diffs) is the known optimization, deferred until profiling demands it. In-repo precedents: `jsonFingerprint` (validate-then-stringify) and `seedToUint32` (literally FNV-1a 32-bit).
- **Post-mortem:** on desync, peers exchange `SessionBundle`s (each peer records locally — the recorder is already per-world) and `diffBundles` pinpoints the divergence (its no-map mode is symmetric with per-tick index alignment — purpose-fit for two peers' independent bundles). Guidance: compare over the overlapping tick range — a veteran-vs-late-joiner comparison walks the union of ranges and reports the non-overlapping head as spurious deltas. The existing AI-first tooling becomes the multiplayer debugging tooling unchanged. This composition — lockstep desyncs debugged with the replay stack — is the design's payoff and why it belongs to this engine specifically.

### 4. What is deliberately NOT designed

Matchmaking, NAT traversal, encryption, cheat prevention beyond determinism (lockstep famously leaks full state to all peers — fog-of-war secrecy needs server-authoritative topology, where `player-observation` (objective 4/7) becomes the per-client filter), and rollback (above).

## Engine gaps inventory (work list when triggered)

1. `world.stateDigest()` (or standalone `digestSnapshot(snap)`) — small, engine-side, independently useful to the benchmark gate and selfCheck as a cheap pre-filter. Must canonicalize component-key order (sort before hashing): component registration order is an ADR-4 contract, not structurally enforced, and two correct peers may differ in `registerComponent` call order — without sorting, that yields a spurious digest desync that the structural `diffBundles` post-mortem would then report as equivalent, a confusing contradiction.
2. `LockstepSession` utility + `LockstepTransport` callback interface — standalone module, ~300 lines, fully unit-testable with an in-memory transport (two worlds, scripted latency/jitter/loss schedules — deterministic tests, no real sockets).
3. Guide with an in-memory two-peer example; ClientAdapter interop notes.

## Test plan (on trigger)

Deterministic in-memory transport suite: barrier under jitter/reorder/loss-with-retry schedules incl. empty-frame delivery; relay-declared leave mid-game (half-open contract: a leave effective at T removes the player from the barrier for T and later on all peers identically; their last required frame is T-1); cross-peer ordering invariance (arrival permutations produce identical worlds — assert via stateDigest); join mid-game incl. version-envelope rejection (exact Node + V8 fingerprint mismatch rejects); **digest round-trip stability** — `digest(serialize(w)) === digest(serialize(applySnapshot-rebuilt w))` and veteran-vs-late-joiner digest equality at the same tick (design-1: selfCheck compares key-order-insensitively, so the existing suite does NOT pin the byte-level JSON stability the digest needs; the late-joiner's first digest exchange is exactly this case); digest mismatch surfaces structured desync with correct tick attribution; composed post-mortem (induced divergence → bundle exchange → `diffBundles` names the tick).
