- High: [src/world.ts:304-324] `destroyEntity()` now calls `this.entityManager.destroy(id)` before running destroy callbacks and before clearing components/resources/tags/meta. That frees the entity ID too early. If an `onDestroy` callback creates a new entity, the free-list can immediately recycle the same ID while the old entity’s cleanup is still pending, and the tail of `destroyEntity()` will then erase the new entity’s state. Even without reuse, callbacks now observe a half-destroyed entity (`isAlive(id) === false` while components/tags/meta still exist), which is a brittle contract change.

- High: [src/world.ts:2203-2237, 946-948] `getDiff()` now advertises a defensive copy, but `cloneTickDiff()` only clones the outer containers. `entry.set.map(([id, data]) => [id, data])` and `state: { set: { ...diff.state.set } }` preserve references to mutable component/state payloads. A caller that mutates the returned diff can still mutate live world data and/or the stored `currentDiff`. The new tests only cover top-level array mutation, so this leak is not exercised.

- Medium: [src/game-loop.ts:33-45] `step()` no longer increments the tick; progress now depends on someone else calling `advance()`. That makes `GameLoop`’s tick semantics callback-coupled and is a breaking API change for any direct `GameLoop` consumer. The diff updates manual-step tests, but there is no coverage here for `start()` / scheduled ticking, so this looks like a likely regression for standalone `GameLoop` usage.
